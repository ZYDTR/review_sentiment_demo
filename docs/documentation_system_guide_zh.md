# 可复用的项目文档写作方法论（通用版）

> 目标：抽象出一套**可迁移到任何项目**的文档写作思路与结构方法。

## 1. 文档体系的“分工法则”
不要用一篇文档承载所有信息，而是把它拆成有明确职责的几类文档：

- **PRD（产品需求）**：定义“做什么”和“为什么做”。
- **RFC（技术方案）**：定义“怎么做”和“为什么这样做”。
- **API/事实参考**：外部接口与约束的真实事实。
- **WORKING（执行日志）**：记录进度、问题、决策与修复。
- **Lessons（经验沉淀）**：把经验固化为可复用规则。
- **专题文档**：当子领域变复杂时拆出独立文档。

**原则**：一类文档只解决一类问题，降低混杂与反复。

## 2. 文档协作的“闭环逻辑”
一套好文档不是静态的，而是闭环：

1. **PRD**：先定义目标、范围、用户与行为。
2. **RFC**：把目标转化成技术方案与约束。
3. **API/事实**：验证外部依赖是否真实可用。
4. **WORKING**：在实施中记录变化与风险。
5. **Lessons**：把过程反思转化为下一次可复用的方法。

结果：文档不仅指导实现，还能持续纠错和升级。

## 3. PRD 的通用写法（产品文档模板思路）
目标：让读者清楚“这个产品是什么、怎么用、边界在哪”。

通用结构：
- **定位与边界**：先写“它是什么”，再写“它不是什么”。
- **用户与场景**：用 2–4 个场景描述真实使用。
- **目标与成功标准**：用可验收的指标表达。
- **功能范围**：必须做 vs 可选。
- **关键交互与行为规则**：细化到用户动作与系统响应。
- **风险与约束**：提前暴露不可控问题。

写作原则：
- 用**具体行为**而非抽象概念。
- 用**默认值和边界**减少歧义。

## 4. RFC 的通用写法（技术方案模板思路）
目标：让工程团队对“设计与实现方式”达成共识。

通用结构：
- **元数据**：编号、状态、日期、PRD 链接。
- **摘要 / 背景 / 目标 / 约束**。
- **架构图**：系统组成与数据流。
- **技术选型 + 理由**：有对比，说明取舍。
- **数据模型 / 状态设计**：核心结构定义。
- **失败策略**：错误、重试、降级。

写作原则：
- 用“为什么这样做”来支撑“怎么做”。
- 关键技术点写清楚边界与风险。

## 5. API/事实文档的通用写法
目标：让所有人对外部依赖有一致理解，避免“猜 API”。

结构要点：
- **来源说明**（官方/调研/测试）。
- **启动方式与鉴权**。
- **按模块分类的接口表**：Method/Path/用途/响应。

写作原则：
- 事实优先，避免推测。
- 能表格就表格，利于检索。

## 6. WORKING 文档的通用写法
目标：让项目执行可追踪、可复盘。

结构要点：
- **当前状态**（日期、阶段、build/test）。
- **进行中 / 已完成 / 待办**。
- **问题记录**：原因 + 修复方式。
- **决策记录**：为什么这样选。

写作原则：
- Checklist 化，进度清晰。
- 记录过程中的真实变化，而非“理想计划”。

## 7. Lessons 文档的通用写法
目标：把经验固化成可复用的方法。

推荐格式：
- **场景** → **错误做法** → **正确做法** → **Lesson**。

写作原则：
- 具体、可操作，而不是口号。

## 8. 文档写作的共通风格
- 标题短、层级清晰。
- 关键点加粗。
- 表格与 ASCII 图提升可读性。
- 默认值与边界条件必须显式写出。
- 尽量“可执行”，而不是“描述性”。

## 9. 文档如何一步步长成（通用方法论）
一个项目的文档不是一次性完成，它会经历：

1. **最小 PRD**：先保证方向清晰。
2. **最小 RFC**：明确技术路径，不阻塞执行。
3. **补 API/事实**：边验证边修正。
4. **执行中记录**：把真实进展写进 WORKING。
5. **专题问题拆文档**：复杂问题拆出来单独管理。
6. **提炼 Lessons**：把走过的坑转成方法。

核心原则：不追求一次完美，而是每一步都“可用 + 可复盘”。

## 10. 如何在任何项目里实践这套方法
- 从 **PRD / RFC / API / WORKING / Lessons** 五类起步。
- 强制写清“是/不是”，避免范围漂移。
- 关键结论必须有事实验证或测试支撑。
- 决策与权衡必须留痕，减少反复争论。
- WORKING 建议每日更新 1–2 次，只写高价值变化。
- 每个阶段补 1–2 条 Lessons，逐步沉淀方法。
